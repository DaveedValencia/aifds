AIFDS ▸ v1 Draft

AI Feed Discovery Standard Specification

Authoritative rules for how AIFDS defines “AI-ready structure.” This draft covers the governance model, scoring rubric, and schema requirements that keep business facts interpretable, verifiable, and trustworthy for AI systems.

1. Scope and Principles

AIFDS is the open standard for structured data governance. It extends schema.org with deterministic rules that align technical accuracy with business truth. This specification defines the minimum expectations for validation, scoring, remediation, and publication.

  • Interpretability: Data must map to schema.org classes and properties in a way that AI systems can reason about unambiguously.
  • Verifiability: Each fact must carry evidence or provenance signals so downstream agents can confirm fidelity.
  • Trustworthiness: Conflicts, duplication, and missing essentials must be surfaced and resolved before distribution.
  • Freshness: Payloads must reflect the current state of the business and expose timestamps that allow audit trails.

2. Governance Structure

2.1 Rule Classes

  • Required — Must pass for a payload to be considered AIFDS compliant.
  • Recommended — Strongly advised; failure lowers the score but does not block compliance.
  • Advisory — Informational guidance based on best practices or emerging patterns.

2.2 Change Management

  • Major updates require an RFC with consensus and version bump (v1.x → v2.0).
  • Minor updates extend the v1 line (v1.0 → v1.1) and must remain backward compatible.
  • Patch updates (v1.0.0 → v1.0.1) address clarifications or errata.
  • Every change logs affected rules, test cases, and schema revisions.

3. Scoring Rubric

AIFDS issues a composite grade of Bad, Average, Good, or Excellent. Each dimension is evaluated qualitatively; the final grade reflects an aggregate judgment with Identity & Trust and Structured Data Completeness treated as critical inputs.

Dimension Signals Failure Modes Priority
Identity & Trust Organization metadata, canonical URLs, contact references, proof of ownership Missing canonical entity, conflicting brand names, unverifiable contact paths Critical
Structured Data Completeness Required schema fields, relationships, cross-entity references Incomplete core fields, orphaned references, schema.org misuse Critical
Navigation & Breadcrumbs Hierarchical breadcrumbs, intra-site linking, canonical navigation Looping breadcrumbs, misaligned canonical paths, missing hierarchies High
Freshness & Evidence Timestamps, change logs, evidence links, corroborating documents Stale timestamps, unverifiable claims, inconsistent update cadence High

Scoring Bands:

  • Excellent — Fully compliant with required rules; only advisory notes remain.
  • Good — Minor recommendations present, no critical gaps.
  • Average — Several high-priority issues or missing recommended fields that limit reliability.
  • Bad — Fails required checks or contains unresolved conflicts; not AI-ready.

4. Validation Workflow

  1. Generate JSON-LD payloads that align with the forthcoming AIFDS schemas (coming soon).
  2. Run schema validation using JSON Schema Draft 2020-12 compatible tooling with the AIFDS rule files.
  3. Execute semantic checks:
    • Cross-reference canonical IDs, URLs, and references across payloads.
    • Detect duplicate or conflicting facts across multiple JSON-LD nodes.
    • Verify evidence sources (Linked documents, canonical pages, or API references).
  4. Produce a validation report with:
    • Overall grade and dimension scores.
    • Rule-level pass/fail with severity.
    • Remediation guidance and suggested evidence anchors.
  5. Publish payloads and reports to partners, ensuring lastModified values track change cadence.

5. Schema Requirements (v1)

Each schema extends schema.org to satisfy AIFDS rule coverage. Required fields must be present and truthful; recommended fields increase score; advisory fields capture optional context. Browse the draft library and example payloads at /docs/schemas/.

5.1 Organization & Website

  • File: business.json and homepage.profile.json (draft)
  • Required: @id, name, url, logo, contactPoint, foundingDate, sameAs
  • Recommended: parentOrganization, hasCredential, awards, serviceArea
  • Advisory: brand, purpose, mission
  • Evidence: Link to legal documents, certification listings, and canonical policy pages.

5.2 Product Schema

  • File: product.json
  • Required: @context, @type, sku, gtin/isbn/mpn, name, description, brand, offers, lastModified
  • Recommended: isSimilarTo, isRelatedTo, review, aggregateRating
  • Advisory: speakable, audience, productCertification
  • Evidence: Product documentation, pricing terms, official photos/videos.

5.3 Services Schema

  • File: services.json
  • Required: @id, serviceType, areaServed, provider, offers/hasOfferCatalog, lastModified
  • Recommended: workExample, deliverable, serviceOutput, serviceLevelAgreement
  • Advisory: customerType, category, industry
  • Evidence: Case studies, proposal templates, third-party reviews.

5.4 Policy & FAQ

  • Files: faqs.json, policies.json
  • Required: Canonical URLs, dateModified, mainEntity Q/A pairs or policy statements.
  • Recommended: Evidence references, authoritative author descriptors, change history.
  • Advisory: audience tags, applicableLocation.
  • Evidence: Signed policy documents, regulatory filings, official press releases.

5.5 Navigation & Breadcrumbs

  • File: navigation.json (forthcoming)
  • Required: Sitewide canonical tree via BreadcrumbList with itemListElement ordering.
  • Recommended: spoke/hub semantics, cross-site @id alignment.
  • Advisory: Alternate language breadcrumbs, mobile-specific adjustments.
  • Evidence: CMS navigation exports, sitemap URLs.

5.6 Freshness Signals

  • Applies to all payloads.
  • Required: lastModified (ISO 8601), inLanguage, contentLocation when relevant.
  • Recommended: version, isBasedOn references, temporalCoverage.
  • Evidence: Republish history, release notes, or changelog links.

6. Remediation Guidance

Conflict Resolution

  • Deduplicate JSON-LD blocks; prefer a single canonical source per entity.
  • When conflicts exist, mark deprecated payloads with expires and reference the canonical @id.

Evidence Anchoring

  • Attach evidence URLs via citation or isBasedOn to support claims.
  • For offline evidence, provide downloadable PDFs or third-party verification tokens.

Audit Trails

  • Log validation runs with timestamps, grades, and change summaries.
  • Expose diff reports showing field-level updates between payload versions.

7. Publication & Distribution

  • Publish payloads via HTTPS endpoints, signed feeds, or structured repositories accessible to partners.
  • Always pair payloads with the latest validation report and scorecard summary.
  • Use stable @id references and versioned URLs so consumers can cache and diff efficiently.
  • Notify consumers of breaking changes via the AIFDS change log and RFC process.
